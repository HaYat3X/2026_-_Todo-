Gitによるバージョン管理とブランチ運用

なぜゲーム開発にGitが必要なのか？
対象：新入社員 テーマ：Git ブランチ運用

01 バージョン管理とGit

ゲーム開発は、数万個のファイルが複雑に絡み合います。Gitは、それらの「いつ・誰が・何を変えたか」をすべて記録し、必要があればいつでも過去の状態に戻せる「タイムマシンのような貯蔵庫（リポジトリ）」を提供します。

⚠️ Gitがないとどうなるか？

・「クエストデータ_最新版2.json」のようなファイルだらけになる

・バグ修正中に他の誰かが同じファイルを書き換えて、データが壊れる

・「昨日まで動いていた演出が急に消えたが、原因が不明」という事態に陥る

02 基本操作と「Ignore」の重要性

操作 説明
Commit 「1面をクリアした」時点でのセーブデータを作る感覚。変更をローカルに記録します。
Push 手元のセーブデータをクラウドに保存し、チーム全員が使えるようにします。
Pull 他のエンジニアやデザイナーが更新した最新のモデルや背景データを取り込みます。

.gitignore の重要性と「絶対にPushしてはいけないもの」

Gitは優秀な記録係ですが、「記録してはいけないもの」まで共有サーバーに送ってしまう危険性があります。これを防ぐのが .gitignore という設定ファイルです。

🚨 最悪の事故：機密情報の漏洩（パスワード・APIキー）

データベースのパスワードやAPIのシークレットキーが書かれた設定ファイルを、誤って一度でもPushしてしまうと、後からファイルを消してもGitの過去の履歴から丸見えになってしまいます。

これを悪意のある第三者に見られると、ゲームのサーバーが乗っ取られたり、顧客情報が流出したりする大惨事に直結します。

プロジェクトに参加したら、まずは .gitignore に「パスワード等の機密情報が含まれるファイル（.envなど）」「大容量のキャッシュファイル」「個人のPC環境設定ファイル」がしっかり指定されているかを確認することが鉄則です。

03 なぜ「ブランチ」を分けるのか？

Gitの最大のメリットは、履歴を枝分かれ（ブランチ）させて並行作業ができることです。ゲーム開発でよくある「絶望的な状況」を例に比較してみましょう。

🎮 事例：新イベント開発中に、既存ボスに重大なバグ発覚！

あなたは今、来週公開予定の「期間限定クエスト」のコードを書いています（まだ50%しか完成しておらず、バグだらけの状態）。そこへ「現在公開中のボスが、特定条件で動かなくなるバグが見つかった！今すぐ直して配信して！」と緊急連絡が入りました。

❌ ブランチがない場合（1本道開発）

手元のコードには「作りかけのイベント」と「ボスのバグ修正」が混ざっています。バグ修正だけを配信したくても、未完成でバグだらけのイベントデータまで本番環境に送られてしまい、ゲーム全体が壊れます。

✅ ブランチがある場合（並行開発）

「イベント開発用ブランチ」はそのまま置いておき、「本番用ブランチ」に切り替えてバグだけを修正・配信できます。完了後、またイベントブランチに戻れば中断したところから安全に再開できます。

04 現場のブランチ戦略

main          ← 本番配信中（Ver 1.0）
└── develop    ← 次バージョン統合・検証（Ver 1.1）
    └── feature/*
        ├── feature/new-character
        ├── feature/gacha-animation
        └── feature/boss-bugfix
ブランチ 役割
main ユーザーが現在遊んでいる本番データ。直接コミット禁止。絶対に壊してはいけない聖域。
develop 次回リリースに向けてみんなの作業を合流させる場所。テストプレイはここで行います。
feature/* 「武器の追加」「ガチャ演出の実装」など、個人の作業ブランチ。完了後 develop へマージ。
運用ルール

作業は必ず develop から feature ブランチを切って進める。main への直接コミットは禁止。マージ前にチームへ一声かけること。

05 Gitを用いた標準的な開発サイクル

実際の開発現場では、ブランチの仕組みを使って以下のような流れで作業を進めます。

1
最新状態の取得（Pull）

作業を始める前に、リモートの最新の develop ブランチからコードを取り込みます。

2
作業ブランチ作成（Branch）

「新キャラ実装」など、自分のタスク専用の feature ブランチを作成します。

3
実装と保存（Commit）

コードやアセットを追加し、キリの良いところでこまめにローカルへセーブを残します。

4
リモートへ反映（Push）

ローカルでの作業内容をリモートリポジトリにアップロードします。

5
レビューと統合（Merge）

チームメンバーに確認（Pull Request）を依頼し、承認されたら develop ブランチに合流させます。

ブランチは「保険」です。恐れずに切って、安全に開発を進めましょう。
