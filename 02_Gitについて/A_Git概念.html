Gitとは

バージョン管理の基礎と実践ワークフロー

バージョン管理の定義

変更履歴の完全な保存
全ての修正をタイムスタンプ、作成者、変更内容と共に記録する。
過去の状態への復元
不具合発生時や仕様変更時に、任意の時点まで確実に立ち戻れる。
非同期的な共同作業
複数の開発者が同一ソースコードに対して並行して変更を加えることを可能にする。

Gitの概要と特性

分散型バージョン管理システム

各開発者のローカル環境にリポジトリの完全なコピーを持つ「分散型」アーキテクチャを採用している。

オフライン対応
通信環境に依存せず高速に動作する。
高信頼性
各所にデータが分散されるため、消失リスクが極めて低い。
高度な分岐
柔軟なブランチ操作による並行開発が可能。

Git不在時のリスクと課題

管理の煩雑化
日付や「最終版」といった手動命名による管理は、確実な履歴追跡が困難になる。
変更の競合と消失
他者の変更を誤って上書きし、貴重な作業内容を消失させるリスクが常に存在する。
復旧コストの増大
手動管理ではエラー発生時の原因特定と復旧に多大な時間を要する。

リポジトリの構造

データの保存場所

リポジトリは、ファイルとその変更履歴を一括して管理する「貯蔵庫」。

ローカルリポジトリ
個人のPC上の履歴管理領域。手元で作業しながら随時コミットできる。
リモートリポジトリ
サーバー上の共有履歴管理領域。チーム全員が参照・更新できる。

主要な操作：Commit、Push、Pull

Commit
ローカルでの変更を履歴として確定。変更意図を記述するメッセージを添える。
Push
ローカルの履歴をリモートリポジトリへ反映し、チームに共有する。
Pull
リモートの最新履歴を取り込み、ローカル環境を同期・更新する。

BranchとMerge

Branch（ブランチ）
メインの履歴から分岐し、他者の影響を受けずに機能開発や修正を行う仮想的な作業領域。
Merge（マージ）
分岐させたブランチの成果をメイン履歴に統合する。競合が発生した場合はGitが検知し解決を促す。

Ignore（.gitignore）の役割

管理対象からの除外
設定ファイル、一時キャッシュ、機密情報など、リポジトリに含めるべきでないファイルを定義する。
品質とセキュリティの担保
不要なファイルの混入を防ぎ、リポジトリの健全性と安全性を維持する。

標準的な開発サイクル

Synchronize
最新のソースをPullで取得しベースを最新化
Isolation
作業用のBranchを作成し開発に着手
Commit
機能ごとにこまめに変更を確定
Publish
検証後、Mergeを経てPushし共有
